# codex-feishu 产品化迭代方案

生成时间：2026-05-23

## 当前状态

codex-feishu 现在已经不是单一脚本，而是一个运行中的私有机器人产品雏形：

- 云端主机：Ubuntu 24.04，工作区由部署环境指定。
- 主服务：`cc-connect.service`，当前 active，运行 patched `cc-connect`。
- 入口：飞书私聊、项目群、汇报群、长期协作群等固定工作区。
- 当前项目数：由部署环境决定的多个 Feishu project
- 辅助服务：可选 fallback Responses proxy、`codex-balance-rotate.timer`、`codex-failure-watchdog.timer`
- 可与其它聊天桥接服务共存，但配置和 systemd unit 必须隔离。
- 核心能力：群消息监听、@ 深度任务、流式卡片预览、画图命令、知识库与本地文件检索、Codex key 轮询和失败切换

当前最像产品的部分：多入口、多模型、多群隔离、云端常驻、失败自动切 key。

当前最不像产品的部分：缺少统一管理后台、配置仍偏手工、密钥和补丁治理不够工程化、可观测性靠日志、发布回滚靠手册。

## 产品定位

codex-feishu 应定位为“面向固定飞书群组和个人工作流的 AI 运维助理平台”，不是泛用聊天机器人。

第一版可上线卖点：

- 私聊：个人 Codex 助理，能处理长任务、查资料、维护文件。
- 项目群：按群隔离知识库、文件、任务规则。
- 深度入口：普通快速模型与深度模型分工，避免成本失控。
- 运维入口：自动切 key、失败 watchdog、远程部署包。
- 媒体入口：群内 `/画图` 直出图片。

核心承诺：

- 不乱说：有群规则、知识库、权限边界。
- 不乱跑：不同群、不同 App、不同模型隔离。
- 不丢活：任务有状态、异常可追踪、失败能恢复。
- 不泄密：密钥不进包、不进仓、不进群。

## P0：上线前硬化

目标：先把“能跑”变成“可靠跑”。

### 1. 密钥和配置治理

问题：

- 运行态配置中可能存在 provider key 和 Feishu app secret。
- 云端配置通常分散在 cc-connect config、环境文件、Codex config。
- 补丁二进制可能被 `cc-connect update` 覆盖。

动作：

- 立刻轮换已经出现在本机配置/聊天上下文中的 key 和 app secret。
- 本机配置改为只引用 secret 文件或环境变量，禁止明文 key。
- 增加 `scripts/audit-secrets.ps1` / `scripts/audit-secrets.sh`，扫描 `sk-`、`appSecret`、`apiKey`、`OPENAI_API_KEY`。
- 云端 cc-connect config 做 sanitized 快照，真实密钥仍只在服务器。
- 固化 patched `cc-connect` 二进制来源，记录 checksum 和恢复命令。

完成标准：

- 仓库和更新包不含明文密钥。
- 每次部署前自动跑 secret audit。
- 补丁被覆盖后有一条命令可恢复。

边界：

- 不为了“安全感”额外收紧 Codex/cc-connect 的运行沙箱；当前阶段只做密钥、发布包、日志和配置边界治理。
- 如果未来要调整 systemd sandbox、文件权限、网络出口或 Codex sandbox，必须先在测试入口验证 `/help`、文件读取、画图、长任务、模型切换和重启恢复。
- 任何可能影响功能路径的安全配置默认不上线，先记录风险和回滚方式。

### 2. 服务健康检查

问题：

- 当前靠 `systemctl status` 和 `tail` 人工判断。
- `cc-connect` 运行正常，但系统有 failed unit：`aegis.service`。
- 服务器内存总量 1.6GiB，free 低但 available 约 843MiB，容量偏紧。

动作：

- 增加 `scripts/codex-feishu-healthcheck.sh`：
  - `cc-connect.service` active
  - 配置了 fallback Responses proxy 时，该服务 active
  - `projects=9`
  - 最近 10 分钟无连续 API failure
  - 磁盘使用率 < 80%
  - 内存 available > 300MiB
- 每 5 分钟 systemd timer 跑健康检查，异常写入专用 health log。
- 异常时发飞书私聊通知给管理员。
- 修复或显式忽略 `aegis.service` failed 状态，避免误报。

完成标准：

- 不 SSH 也能知道服务是否健康。
- 任一入口失效 5 分钟内通知。

### 3. 发布和回滚自动化

问题：

- 现在有文档流程，但仍多步手工。
- 部署整体替换工作区时，若不重启会出现 deleted cwd 风险。

动作：

- 将现有部署流程固化成发布脚本和服务器应用脚本。
- 每次部署自动：
  - 打包白名单目录
  - 跑 secret audit
  - 上传
  - 服务器备份
  - 原子替换
  - chmod 脚本
  - restart `cc-connect`
  - healthcheck
  - 失败自动回滚
- 生成 `release.json`，记录版本、时间、git commit、包 hash、服务版本。

完成标准：

- 一条 PowerShell 命令完成发布。
- 失败自动回滚到上一个可用版本。

## P1：产品体验

目标：让用户觉得这是一个稳定产品，而不是“群里接了个 agent”。

### 1. 统一命令体系

建议命令：

- `/help`：列出当前群可用能力。
- `/status`：显示当前入口、模型、是否忙、最近一次错误。
- `/new`：新开任务上下文。
- `/stop`：停止当前长任务。
- `/fast`：切快速模式。
- `/deep`：提示用户 @ 深度 bot。
- `/files`：列出本群可用文件。
- `/remember`：写入群记忆。
- `/forget`：按权限删除记忆。
- `/画图`：生图。

动作：

- 每个群的 `INSTRUCTIONS.md` 统一命令语义，只保留群差异。
- 命令尽量在平台层处理，不进入模型，减少延迟和成本。
- 所有命令输出用飞书卡片，不用长纯文本。

完成标准：

- 新用户进群后只需发 `/help`。
- 80% 常用操作不用自然语言猜。

### 2. 卡片化交互

动作：

- 长任务开始：给原消息加 `OnIt`，再发进度卡。
- 进行中：卡片显示当前阶段、已用时间、可取消按钮。
- 完成：卡片给“结果 / 文件 / 继续追问 / 转深度处理”按钮。
- 失败：卡片展示可读错误、重试按钮、管理员诊断 ID。

完成标准：

- 用户能看见任务是否被接住。
- 失败不是静默或堆日志。

### 3. 群工作区产品化

动作：

- 每个群固定目录结构：
  - `AGENTS.md`：角色和安全边界
  - `INSTRUCTIONS.md`：用户可见规则
  - `KNOWLEDGE.md`：群核心知识
  - `memory/`：长期记忆
  - `local_files/`：文件资产
  - `runs/`：任务记录
- 增加 `workspace manifest`：
  - 群名、chat_id、启用入口、模型、允许命令、知识库更新时间。
- `help` 输出直接从 manifest 生成。

完成标准：

- 新增一个群，不再复制粘贴一堆目录后手工改。
- 群能力可被程序读取和展示。

## P2：管理后台

目标：从“SSH 运维”升级成“可管理的 SaaS-like 产品”。

### 1. Web 控制台

第一版页面：

- 总览：服务状态、项目数、最近错误、当前模型、余额状态。
- 项目：9 个 project 的启停、模型、入口、群绑定。
- 会话：正在运行任务、历史任务、失败任务。
- 知识库：每个群的文件、记忆、最近更新。
- 发布：当前版本、上次部署、回滚按钮。
- 密钥：只显示状态和过期时间，不显示明文。

技术建议：

- 后端：FastAPI 或 Go 小服务，仅绑定 `127.0.0.1`，通过 Tailscale/SSH tunnel 访问。
- 前端：React + Vite，静态资源由后端托管。
- 鉴权：管理员 token + IP allowlist，后续接飞书 OAuth。

完成标准：

- 不看 SSH，能完成 70% 日常运维。
- 不暴露公网管理后台。

### 2. 任务队列和审计

动作：

- 每条消息进入任务表：
  - `run_id`
  - 来源群/用户
  - 入口项目
  - 模型
  - 状态
  - 开始/结束时间
  - token/成本估计
  - 错误摘要
- 长任务进入队列，避免同群并发互相抢上下文。
- 高风险工具调用前记录审计项。

完成标准：

- 能回答“昨天谁让 bot 做了什么、花了多少、失败在哪里”。

## P3：能力商业化

目标：从自用运维项目走向可交付产品。

### 1. 模板化行业方案

先做 3 个模板：

- 项目群助理：资料整理、会议纪要、进度日报、代码仓库问答。
- 长期协作群助理：提醒、清单、共享知识、文件查找。
- 竞赛群助理：赛题资料、任务分工、实验记录、周报生成。

每个模板包含：

- 群配置
- 命令菜单
- 默认知识库结构
- 权限策略
- 成本策略
- 示例卡片

### 2. 多租户隔离

动作：

- tenant/project/workspace 三层模型。
- 每个 tenant 独立密钥、日志、知识库、文件存储。
- 严格禁止跨群读取，除非管理员显式授权。

完成标准：

- 可以给第二个团队部署，不担心读到第一个团队数据。

### 3. 可靠性指标

上线指标：

- 月可用性 >= 99%
- 普通命令 P95 首响应 < 3 秒
- 深度任务 10 秒内给进度卡
- 消息漏处理率 < 0.5%
- 自动恢复成功率 > 90%
- 每次发布可回滚

## 推荐迭代节奏

第 1 周：P0

- 轮换泄露密钥
- secret audit
- healthcheck
- 一键部署和回滚
- release.json

第 2 周：P1

- `/help`、`/status`、`/stop`
- 群 manifest
- 统一卡片状态
- 失败可读化

第 3-4 周：P2

- 只读控制台
- 任务表和审计
- 项目启停
- 发布状态页

第 2 个月：P3

- 模板化部署
- 多租户隔离
- 成本统计
- 飞书 OAuth 管理

## 立即建议

下一步优先做这 5 件：

1. 轮换本机配置里出现过的 API key 和 Feishu app secret。
2. 写 `scripts/audit-secrets.ps1`，把部署前扫描变成硬门禁。
3. 写云端 `healthcheck.sh`，检查 `projects=9`、systemd、磁盘、内存、近期错误。
4. 把部署流程封装成一条命令并自动回滚。
5. 为三个群生成 `workspace manifest`，让 `/help` 和控制台有结构化数据来源。
